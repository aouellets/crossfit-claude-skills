# Contributing

Thanks for helping improve the CrossFit coaching skills. This repository is the
**source of truth** for content that downstream catalogs (e.g. Skill Me) ingest,
so contributions must clear the validation and eval gates before merge.

## Ground rules

1. **Coaching content, not medical advice.** Every coaching skill applies the
   [safety-and-referral](references/safety-and-referral.md) framework
   proportionately. Never diagnose, prescribe medication, treat clinical
   conditions, or guarantee outcomes. See [docs/safety-and-scope.md](docs/safety-and-scope.md).
2. **Trademark discipline.** Capitalize CrossFit (capital C, capital F), never
   use it as a verb, and never call the repo, pack, or a skill official,
   certified, or endorsed. See [docs/branding-and-trademarks.md](docs/branding-and-trademarks.md).
3. **No fabricated authority.** Never invent official competition standards,
   rulebook penalties, or certification exam content. Rules-dependent skills
   must enforce versioning (see [references/source-version-registry.json](references/source-version-registry.json)).
4. **Standalone.** This repo has no parent, sibling, or shared-core dependency.
   Do not import from or reference any other repository.

## Local setup

```bash
npm install
npm run validate   # frontmatter + structure + links + triggers + sources
npm run build      # regenerate skill-inventory.json and seed/seed.sql
npm run evals      # run the eval harness
npm run check      # all of the above
```

Node 20+ is required.

## Authoring or editing a skill

Read [docs/skill-authoring-standards.md](docs/skill-authoring-standards.md) in
full first. The short version — every skill folder under `skills/<id>/` needs:

- `SKILL.md` with valid frontmatter (`name`, `description`) and a body covering:
  workflow, required + optional inputs, explicit **non-goals**, a structured
  **default output**, safety/scope rules, incomplete-information handling, and a
  **Related Skills and Routing** section.
- `examples/` with **≥3** worked examples.
- `evals/cases.json` with **≥10** eval prompts **plus ≥1 adversarial case**,
  `evals/rubric.md`, and `evals/expected-behaviors.md`.

The skill `id`/slug is the folder name: `lowercase-hyphenated`, globally unique.

### Description = the trigger surface

Descriptions are how `browse_skills` and `get_active_skills` decide what to
load. Front-load the trigger condition, write in the third person, state **when**
to load, the **user language** that signals it, the **inputs** expected, what it
**excludes**, and **which sibling skill** owns the request when adjacent. Vague
descriptions ("Helps with CrossFit") fail `validate:triggers`.

## Workflow

1. Branch from `main`.
2. Make a small, single-purpose change. Separate refactors from content changes.
3. Run `npm run check` — it must pass with no errors.
4. If you added/edited a skill, confirm `manifest.json` still lists exactly the
   intended skill IDs and that `npm run build` produced a clean diff.
5. Update [CHANGELOG.md](CHANGELOG.md) under `[Unreleased]`.
6. Open a PR describing what changed and why, plus any safety/scope implications.

## Generated files — do not hand-edit

`skill-inventory.json` and `seed/seed.sql` are generated by
`scripts/build-inventory.ts` and `scripts/build-seed.ts`. Edit the `SKILL.md`
frontmatter and `manifest.json`, then run `npm run build`.
